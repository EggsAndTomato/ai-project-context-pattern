# ADR-007：拆分 index 触发与 log 触发（log 覆盖所有结构/意图变更）

> 日期：2026-06-20  状态：已采纳

## 背景
原 CRITICAL 规则把两件事焊在一条里：「在 wiki/ 下创建任何新文件后 → index +1 行、log +1 条」。这造成两个问题：

1. **log scope 与触发条件冲突**：`log.md` 头部自述 scope 是"结构/意图层面的变了什么 + 为什么"（按**变更性质**，范围宽），但规则触发条件是"wiki 下创建新文件"（按**文件事件**，范围窄）。LLM 跟权威行为规范（CRITICAL），于是系统性采用窄定义、少记。
2. **盲区**：根级文件（README、方法论文档、AGENTS.md 规则本身）不在 `wiki/` 下，新建/大改都不触发；编辑现有文件（即便改的是核心术语/决策）也不触发。一次把核心措辞从"编译一次"改为"增量沉淀"的变更因此没被记入 log——正是这条规则的直接后果。

这违背 ADR-003 的前提：CRITICAL + 工具动作级规则的价值在于**无歧义**，而这条规则在"何时 log"上恰恰留了歧义。

## 选择
把原单条拆成两条，让 index 与 log 各自回归本来的 scope：
- **index 触发**（内容目录、wiki 作用域、文件粒度）：仅在 `wiki/` 新增页面后 → index +1 行。
- **log 触发**（事件流水、全项目作用域、变更粒度）：发生结构/意图层面的变更（新建 / 删改 / 改名文件、根级文档修订、核心术语或决策调整）后 → log +1 条。

## 理由
1. **同质才该共用触发**：index 是"页面目录"（文件粒度），log 是"事件流水"（变更粒度），二者 scope 本就不同；强行焊死等于用 index 的窄 scope 绑架 log。
2. **消除歧义**：明确 log 的触发是"变更性质"而非"文件事件"，与 `log.md` 自述 scope 对齐，LLM 不再有窄/宽两套定义可选。
3. **堵住盲区**：显式覆盖根级文件修订与"改现有文件"两类此前漏记的情况。
4. **守住 ADR-003 前提**：规则越精确，遵循度越高（ADR-003 的核心论点）。

## 后果
- log 会比以前更频繁地写（任何结构/意图变更都记）——符合其定位，且单条成本极低。
- "结构/意图层面"仍是意图判读词，边界靠 `log.md` 头部已声明的"琐碎代码修订走 git"这条互补规则界定：纯代码小改不记，结构性改动才记。
- 需在 `AGENTS.md` 与方法论文档同步维护这条规则（已同步）。
