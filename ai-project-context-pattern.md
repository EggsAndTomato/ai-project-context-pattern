# AI Wiki 上下文模式（LLM 自维护的项目知识库）

> 适用于用 AI（opencode、Claude Code 等）做 vibe coding 的个人 / 小团队。
> **本文档本身给 AI 读**：在新项目里，让 AI 读完本文后按"第八节"搭建项目知识库。
> 核心目标：把项目知识**编译一次、持续维护**，而不是每次对话重新发现。

---

## 一、核心思想：编译一次、持续维护（反 RAG）

常见做法是 RAG：上传一批文档，每次提问时检索相关片段、现场拼答案。问题是**每次都从零重新发现**，知识不累积——问一个需要综合五份资料的问题，AI 每次都得重新找、重新拼，什么都不沉淀。

本模式不同：**LLM 增量地构建并维护一个持久 wiki**——一组结构化、互链的 markdown，夹在你和原始信息之间。新知识进来时，AI 不只是把它索引好等以后检索，而是**读它、提炼要点、整合进既有 wiki**：更新相关页、修订综合、标注矛盾、补交叉引用。知识**编译一次，然后保持最新**，而不是每次提问重算。

关键差别：**wiki 是一个会复利的产物。** 交叉引用已经建好、矛盾已经标出、综合已经反映你读过的一切。每加一份资料、每问一个问题，wiki 都变得更丰富。

分工：**人负责 sourcing、探索、问对问题；LLM 负责所有 bookkeeping**（总结、交叉引用、归档、保持一致性）——正是这些事让人放弃维护知识库，而 LLM 不会嫌烦。

## 二、结构

| 层 | 位置 | 加载方式 | 内容 | 大小目标 |
|---|---|---|---|---|
| 基础层 | `./AGENTS.md`（**项目根**） | 自动加载 | CRITICAL 维护规则 + 1 行定位 + **index（全量目录）** | ≤ ~300 行（超了就精简） |
| 揭示层 | `wiki/*.md` + `log.md` | AI 按需 Read | 项目知识 + 变更日志 | 不限，靠 index 导航 |

```
项目根/
├── AGENTS.md            ← 基础层（自动加载）
├── log.md               ← 项目变更日志
└── wiki/
    ├── architecture/    ← 固定：技术架构
    ├── decisions/       ← 固定：ADR（为什么这么定）
    ├── bugs/            ← 固定：bug 复盘
    ├── sessions/        ← 固定：会话归档
    └── *.md             ← 其余：概念 / how-to / 领域知识 / 参考笔记（LLM 自由建、互链）
```

> `wiki/` 下**只允许这 4 个子目录 + 根级 loose 文件**，禁止新建其他子目录。一切页面都要进 AGENTS.md 的 index。本模式**不设 raw 层**（不需要不可变原始资料的隔离区）。

---

## 三、基础层：AGENTS.md

### 必须放项目根

opencode / Claude Code 只自动加载：项目根的 `AGENTS.md`（及向上遍历父目录）、全局 `~/.config/opencode/AGENTS.md`。**子目录里的 AGENTS.md 不会被加载**。放错位置 = 整个模式失效。

### 内容 = 规则 + 定位 + index

AGENTS.md 由三块组成：

1. **顶部：CRITICAL 维护规则**（工具动作级，见第七节为什么这么写）
2. **项目定位**：1 行（含技术栈关键词）
3. **index**：wiki 全量目录，按 4 个固定类目 + loose 分组，每条一行摘要

> index **永驻 AGENTS.md、不外置**。外置 = 把"导航中枢"变成懒加载，违背本模式前提。index 膨胀了就**精简**（合并碎条、归档过时项、缩短摘要），不要切成懒加载。~300 行作为一次"目录瘦身 Lint"的触发线。

### AGENTS.md 模板

```markdown
# {项目名}

> 定位：{一句话，含技术栈}。

<!-- ============ 维护规则（CRITICAL）·人定，LLM 禁改 ============ -->
## 维护规则（CRITICAL）

- 回答任何实质问题前：你的**下一个动作**是完整阅读下方 index，再决定读哪些 wiki 页。
- 在 wiki/ 下**创建任何新文件后**：你的**下一个动作**是在下方 index 追加一行，并在 `log.md` 追加一条。
- wiki/ 下**只允许** 4 个子目录（architecture/decisions/bugs/sessions）或根级 loose 文件；**禁止新建其他子目录**。
- 你**只能修改本文件的 index 区**；本文件其余区域（规则、定位）禁止修改。
- **禁止归档原始会话**；session 归档前必须先总结。

<!-- ============ 项目定位·人定 ============ -->
## 项目定位
{一句话定位，含技术栈}

<!-- ============ index·LLM 维护 ============ -->
## index（wiki 全量目录）
### architecture
| 文件 | 摘要 |
|---|---|
| `wiki/architecture/overview.md` | {一句话} |

### decisions
| 文件 | 摘要 |
|---|---|
| `wiki/decisions/001-xxx.md` | {一句话} |

### bugs
| 文件 | 摘要 |
|---|---|

### sessions
| 文件 | 摘要 |
|---|---|

### 其他（根级 loose）
| 文件 | 摘要 |
|---|---|
| `wiki/{concept}.md` | {一句话} |
```

## 四、揭示层：wiki/ 与 log.md

- **4 个固定子目录**装"有模板的结构化产物"；**根级 loose 文件**装其余知识（概念、how-to、领域笔记、参考）。靠 index 分类导航 + 双链互连。
- LLM 自主维护：创建页、更新页、补交叉引用、保持一致性。
- **单一事实源**：同一知识只在一处权威表述，别处只链接，不复制。
- `log.md` = 项目变更日志（结构、意图层面的"变了什么 + 为什么"，最新在上；琐碎代码修订走 git）。

### 固定子目录与页面模板

**`architecture/` — 技术架构页**
```markdown
# {组件/系统名}

> 边界：本文写"是什么 + 怎么搭"。

## 职责
{这个组件干什么}
## 组成与数据流
{模块、关键路径、数据走向}
## 边界与接口
{对内对外的契约}
## 关键约束
{不可违反的硬约束}
## 相关
- 决策依据：`decisions/xxx.md`
- 运行细节：（按需链接其他页）
```

**`decisions/` — ADR（为什么这么定）**
```markdown
# ADR-{编号}：{决策标题}

> 日期：{YYYY-MM-DD}  状态：已采纳 / 已废弃 / 已替代

## 背景
{为什么要做这个决策；候选方案}
## 选择
{选了什么}
## 理由
{为什么选它}
## 后果
{代价、影响、被牺牲的取舍}
```

**`bugs/` — bug 复盘**
```markdown
# BUG-{编号}：{症状一句话}

> 日期：{YYYY-MM-DD}  状态：已修复

## 症状
{表现、触发条件、影响}
## 排查思路
{走过哪些方向、怎么定位的}
## 根因
{真正的原因}
## 方案
{怎么修的；防止复发的措施}
## 复盘
{教训、是否暴露架构问题、关联决策}
```

**`sessions/` — 会话归档（总结后）**
```markdown
# SESSION-{日期}-{主题}

> 日期：{YYYY-MM-DD}  归档于：{YYYY-MM-DD}

## 主题
{这次会话在解决什么}
## 关键决策
{拍板了什么；沉淀到哪些页/ADR}
## 结论与产出
{代码/页面变更摘要}
## 下一步
{未完成、待办}
```

## 五、三操作：Ingest / Query / Lint

**Ingest（纳入新知识）。** 新知识进来（读了份资料、搞懂了个概念、定了个决策）→ LLM 写/更新相关页 → **更新 index** → **追加 log**。一次 ingest 可能触碰多页。

**Query（提问）。** 回答问题前先扫 index 定位相关页，再读、再综合，带引用。**好答案可以归档为新页**——你做过的一次对比、一次分析、发现的一个联系，别让它消失在聊天里，让它继续复利。

**Lint（自愈网·必需）。** 定期让 LLM 体检 wiki，并修：
- 页面之间互相矛盾
- 孤儿页（没有任何入链）
- index 缺条目 / index 有条目但文件已不存在
- 过时主张（被新知识取代但没更新）
- 重要概念被反复提到却没有自己的页
- 缺失的交叉引用
- AGENTS.md 超过 ~300 行 → 做一次目录瘦身

> Lint 是**必需**而非可选。因为"更新 index/log"这类行为指令一定会偶尔被违反（见第七节），Lint 就是对"不完美遵循"的自愈网——发现漂移并修复。

## 六、会话归档纪律

`sessions/` **按需归档**，不是每次都存：
- **用户主动提出归档**，或
- **LLM 判断本次会话重要，提示用户是否归档**。

归档**前必须先总结**（按 sessions 模板提炼：主题/关键决策/结论/下一步），**原始会话记录不存**。重要决策应同时沉淀到 `decisions/`（ADR）和 `log.md`，session 页只保留"那次会话发生了什么"的脉络。

> 这等于是把"会话成果沉淀纪律"落进一个固定目录，而不是禁止归档——给按需总结后的会话一个正式归宿。

---

## 七、为什么规则用 CRITICAL + 工具动作级

经验事实：**"载入上下文" ≠ "被遵循"。** 平淡的行为规则（尤其需要判读用户意图的，如"询问时别改代码"）经常被违反——模型的"乐于助人"默认会压过一条不显眼的规则。

由此得出三条设计对策：

1. **别把存亡攸关的事交给"行为合规"，要让"正确动作"成为"零成本默认动作"。**
   → 所以 index 放进 AGENTS.md：读 index 不是一条要遵守的指令，而是**它本来就在眼前**，导航近乎兜底。
2. **把维护规则写成工作流里的具体"下一个动作"，而不是抽象口号。**
   → ❌「你必须保持 index 最新」；✅「创建 wiki 文件后，你的**下一个动作**是追加 index 一行 + log 一条」。约束工具动作的规则，远比约束意图判读的规则耐破。
3. **承认会漏，所以 Lint 必需。** 行为指令一定会偶尔被违反；Lint 定期扫漂移并修复，是对不完美遵循的对冲。

## 八、给 AI 的 Bootstrap 流程（本文档的用法）

本文档是给 AI 读的。在新项目里，AI 读完本文后按以下步骤搭建知识库：

1. **摸底**：读代码库；必要时问用户——定位、技术栈、已有架构与决策。
2. **建基础层**：在**项目根**创建 `AGENTS.md`（CRITICAL 规则 + 1 行定位 + index 骨架）。
3. **建揭示层骨架**：建 `log.md`（空骨架）+ `wiki/` 下 4 个固定空子目录。
4. **填 architecture 初版**：至少写一个 `wiki/architecture/overview.md`，并在 index 登记。
5. **自检**：用第九节检查清单。

> 原则：先搭骨架，知识随用随补；index 必须和实际同步。

---

## 九、检查清单

| 检查项 | 说明 |
|---|---|
| AGENTS.md 在**项目根**？ | 子目录的不自动加载 |
| AGENTS.md 顶部有 CRITICAL 维护规则？ | 工具动作级、显眼 |
| index 永驻 AGENTS.md、未外置？ | 外置即自毁前提 |
| index 与 wiki 实际一致？ | 无缺条 / 无幽灵条 |
| `wiki/` 下只有 4 个固定子目录 + 根级 loose？ | 禁止其他子目录 |
| 4 个固定目录都存在（哪怕空骨架）？ | 架构完备 |
| 每个固定页用对应模板？ | 结构一致 |
| 没有 raw 层？ | 本模式不设 raw |
| 会话按需、总结后归档？ | 不存原始会话 |
| Lint 定期跑？ | 自愈网 |
| 单一事实源？同一知识只一处权威？ | 别处只链接 |
| AGENTS.md ≤ ~300 行？ | 超了做瘦身 Lint |

## 十、反模式（合集）

1. **AGENTS.md 放错位置**（非项目根）→ 不自动加载，模式失效
2. **index 外置懒加载** → 导航中枢不可见，自毁前提
3. **wiki/ 下自建顶级目录** → 破坏固定结构，去 4 固定类目之外找地方放
4. **设 raw 层** → 本模式不需要；资料就是 wiki 内容
5. **原始会话全量存档** → 无限增长且腐化；改为总结后按需归档
6. **三级目录树** → AI 靠 index+标题导航，不靠文件夹深度；用扁平文件 + 模板
7. **依赖行为指令、不做 Lint** → 行为规则必被偶尔违反；Lint 是必需的自愈网
8. **同一知识多处复制** → 违反单一事实源，必然漂移
